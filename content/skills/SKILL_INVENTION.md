## Purpose

Use this skill whenever Dylan asks for project ideas, app ideas, website ideas, portfolio concepts, creative coding projects, AI-buildable products, weird tools, side projects, or anything that sounds like:

* “Give me ideas for apps I could build”
* “What should I make next?”

* “I want to build something weird/useful/fun”

The goal is not to produce generic startup sludge. The goal is to invent project ideas that feel like Dylan could plausibly make them, care about them, write about them, and add them to Sandwich Codes without quietly dying inside.

## Core invention principles

### 1. Combine personal relevance with buildability

Every idea should feel like it comes from Dylan’s actual world: creative hobbies, product background, Melbourne context, portfolio goals, and strange little obsessions.

Bad:

> A task management app for creatives.

Better:

> A “creative project weather report” that turns your unfinished music, writing, and coding projects into a playful forecast: high pressure from guilt, scattered showers of half-finished MIDI files, chance of actually finishing something by Sunday.

### 2. Prefer specific tools over vague platforms

Avoid broad concepts like:

* “A social network for artists”
* “A marketplace for musicians”
* “A productivity platform”
* “An AI assistant for creators”

Prefer tight, buildable tools:

* “Paste in a poem and generate three melody contour options”
* “Upload a DJ setlist and get a visual map of energy, genre drift, and chaos”
* “Turn photos from a walk in the Dandenongs into a deterministic colour palette and ambient loop”
* “Generate LEGO mosaic instructions from an abstract image using only bricks the user owns”

### 3. Make the demo obvious

Each idea should have a clear “show me” moment.

A good demo can be:

* Upload file → receive surprising output
* Type text → get playable/visual result
* Drag things around → export something useful
* Select constraints → generate artefact
* Compare before/after
* Reveal hidden structure in media, text, colour, sound, or data

If the idea cannot be demoed in under 60 seconds, simplify it.

### 4. Aim for “small, complete, polished”

Dylan’s projects should feel like finished artefacts, not underfed SaaS skeletons.

Prefer:

* One excellent workflow
* A delightful output
* Strong visual identity
* Clear export/share/download
* No unnecessary accounts
* Minimal backend
* Vercel-friendly architecture where possible

Avoid:

* Multi-sided marketplaces
* Social graphs
* Enterprise admin panels
* Complex mobile-first requirements unless requested
* Anything requiring a giant labelled dataset
* Anything that depends on expensive APIs unless there is a free/local fallback

### 5. Use AI deliberately, not automatically

Do not make everything “AI-powered” by default.

Consider four possible build modes:

1. **No AI** — deterministic, algorithmic, local, cheap, elegant
2. **Optional AI** — AI improves output but is not required
3. **AI-assisted** — AI handles interpretation, naming, suggestions, summaries
4. **AI-core** — the product only works because of an AI model

Dylan often likes algorithmic generation, deterministic behaviour, and Vercel-friendly builds. If AI is suggested, explain why it is justified.

### 6. Respect privacy and safety

For projects involving causes, identity, location, audio, photos, or personal writing:

* Prefer local processing where possible
* Avoid requiring accounts
* Avoid storing sensitive data
* Make deletion/export obvious
* Do not build tools that infer identity, sexuality, mental health, politics, or vulnerability
* Do not suggest surveillance-like features

### 7. Make it sound like Dylan would actually build it

The tone of ideas should be practical, dry, observant, and slightly odd.

Avoid:

* “Revolutionise”
* “Empower”
* “Democratise”
* “Unlock your potential”
* “Seamless experience”
* “All-in-one platform”
* “Next-generation”
* “AI-powered ecosystem”

Prefer:

* “A small web app that…”
* “The joke is that…”
* “The useful bit is…”
* “This would be fun because…”
* “The MVP is tiny…”
* “The portfolio value is…”

If the user gives constraints, obey them.

Examples:

* “20 app ideas” → produce 20 concise ideas
* “Ideas for Vercel” → include stack/build notes
* “Music projects” → lean heavily into MIDI, audio, lyrics, DJing, streaming metadata
* “Portfolio projects” → include why each would look good on Sandwich Codes
* “Weird ideas” → increase novelty, reduce commercial framing

### Step 2: Build from intersections

Generate ideas by combining:

* One personal interest
* One work/domain context
* One interaction/output mechanic
* One build constraint

Example formula:

> [Interest] + [domain experience] + [playful mechanic] + [small build]

Examples:

* LEGO + advertising + media planning → a block-based campaign planner
* DJing + streaming media + data visualisation → setlist energy mapper
* Poetry + music + MIDI export → verse-to-melody sketcher
* Photography + Melbourne mountains + generative art → walk-to-palette field recorder
* Accounting + video games + product UX → boring finance concepts explained as game mechanics
* Retail + art + ecommerce → shelf layout simulator for imaginary products

Do not always use the same formula. Some ideas should begin from a tension, a rule system, a physical making problem, an awkward professional truth, a tiny game loop, or an interaction pattern rather than a neat input/output transformation.

### Step 3: Filter for portfolio value

Prioritise ideas that are:

* Distinct from existing projects
* Easy to explain
* Visually interesting
* Buildable as a solo project
* Demoable
* Potentially article-worthy
* Aligned with Dylan’s voice and taste
* Not dependent on huge ongoing content moderation or data collection

### Step 4: Score ideas silently

Before presenting ideas, mentally score each one from 1–5 across:

* Personal fit
* Novelty
* Buildability
* Visual/demo appeal
* Portfolio value
* Relevance to existing projects
* Potential for a good write-up

Prefer ideas that score highly across most categories, not ideas that are perfect in one category and terrible everywhere else.

## Breadth and novelty controls

Do not overfit to Dylan’s existing project patterns. The skill should generate ideas that belong in Dylan’s world, but not all ideas need to be:

* creative input → structured output
* data → playful visualisation
* local-first personal utility
* music/LEGO/weather/receipt/backlog themed
* paste/upload something → generate a clever artefact

When generating ideas, deliberately vary the type of project across multiple invention lanes unless the user asks for a specific category. The goal is range: small tools, strange games, interactive arguments, browser experiments, practical professional artefacts, and creative objects that could sit on Sandwich Codes without all using the same skeleton.

### Invention lanes

Use a mix of these lanes unless the user asks for a narrow category:

1. **Practical tool** — solves a small real problem with a polished workflow.
2. **Creative instrument** — lets the user make sound, image, text, layout, rhythm, structure, or visual systems.
3. **Tiny game** — has rules, constraints, scoring, failure, progression, randomness, or replayability.
4. **Interactive essay** — explains an idea through interaction rather than static prose.
5. **Simulation or sandbox** — lets users manipulate a system and observe consequences.
6. **Personal utility** — helps organise, decide, reflect, track, plan, compare, or salvage something.
7. **Public-interest tool** — helps people understand systems, rights, policies, services, risks, or trade-offs.
8. **Art object** — primarily aesthetic, poetic, strange, atmospheric, or experiential.
9. **Professional artefact** — shows product, marketing, retail, measurement, strategy, or research thinking.
10. **Provocation** — makes a point about technology, AI, privacy, creativity, labour, retail, measurement, or culture.
11. **Browser experiment** — uses web APIs, sensors, canvas, audio, files, gestures, drag/drop, or interaction in an unusual way.
12. **Hardware-adjacent tool** — supports 3D printing, LEGO, photography, music gear, MIDI, sewing, painting, or physical making.

### Novelty modes

When brainstorming, silently choose or mix one or more novelty modes:

* **Safe** — practical, buildable, obvious portfolio value.
* **Weird** — memorable, odd, less obviously commercial, but still coherent.
* **Serious** — professional, product-led, research-led, industry-relevant, or analytically useful.
* **Art** — expressive, visual, poetic, atmospheric, or primarily experiential.
* **Game** — interactive, rule-based, replayable, playful, and possibly competitive.
* **Utility** — genuinely useful, even if small or unglamorous.
* **Provocation** — raises a question or critique through the product itself.

Do not default to Safe + Utility every time. Dylan often wants ideas that are buildable, but the portfolio also needs range. A good batch may include something useful, something weird, something beautiful, something game-like, something serious, and something that makes a point.

### Anti-repetition rule

Across a list of ideas, avoid repeating the same base pattern more than three times.

For example, do not produce a list where most ideas are:

* upload file → generate artefact
* paste text → receive critique
* personal data → whimsical metaphor
* weather/receipt/backlog → visual output
* AI assistant for niche creative task
* dashboard-but-cute
* generator-but-with-a-better-name

If several ideas share the same pattern, replace some with games, simulations, interactive essays, strange utilities, professional artefacts, physical-making tools, or browser experiments.

### Emotional truth rule

Strong ideas should often begin from a real tension, not just a data source.

Examples:

* Starting projects is easier than finishing them.
* Creative tools either over-assist or abandon you.
* Product prioritisation pretends to be objective.
* Retail measurement is powerful but often creepy.
* AI makes output cheap but taste more important.
* People want privacy until convenience quietly eats it.
* Hobbies are joyful but also full of admin, guilt, storage problems, and tiny expensive objects.
* Professional work often hides subjective judgement inside fake certainty.
* A beautiful interface can make a boring workflow feel like an object worth keeping.

Prefer ideas that turn one of these tensions into an interaction model. “Receipts become fish” is a mechanic. “My grocery habits are a tiny domestic biography I did not consent to writing” is an emotional truth. Start from the second one when possible.

### Breadth target for lists

When asked for 10 or more ideas, aim for a spread like this unless the user gives tighter constraints:

* 2–3 practical tools
* 2–3 creative instruments or art objects
* 1–2 games or simulations
* 1–2 professional/product/retail/marketing artefacts
* 1–2 provocations or interactive essays
* 1–2 wildcards that are harder to classify

This is a guide, not a spreadsheet wearing a small reflective vest. The point is to stop every idea from coming out of the same mould.

## Output formats

Choose the format based on the user’s request.

### Default brainstorming format

For each idea, include:

1. **Name**
2. **One-line concept**
3. **Why it fits Dylan**
4. **MVP**
5. **Build notes**
6. **Portfolio angle**

Keep each idea concise.

Example:

```markdown
### 1. Brick Weather

A tiny tool that turns Melbourne weather into a LEGO colour palette and abstract build prompt.

**Why it fits:** LEGO, local context, playful data, visual output.
**MVP:** Pull weather data, map conditions to colours/shapes, export PNG build card.
**Build notes:** Vercel serverless + weather API + deterministic rendering.
**Portfolio angle:** Shows data-to-art thinking without becoming another dashboard in a trench coat.
```

### Compact list format

Use this when the user asks for many ideas.

```markdown
| # | Idea | Concept | Why it fits |
|---|------|---------|-------------|
| 1 | Name | Short description | Fit rationale |
```

### PRD-ready shortlist format

Use this when the user wants stronger ideas, not just quantity.

For each idea:

* Name
* Tagline
* User problem
* Core loop
* MVP features
* Technical approach
* Risks
* Why this belongs on Sandwich Codes

### Top 5 recommendation format

Use this when ideas need judgement.

For each idea:

* Name
* What it is
* Why it is strong
* What to build first
* Why it beats the others

End with:

```markdown
My strongest recommendation: [name]

Because: [brief product/design rationale]
```
